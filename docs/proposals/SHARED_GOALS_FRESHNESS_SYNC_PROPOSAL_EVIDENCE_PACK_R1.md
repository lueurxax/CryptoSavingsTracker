## H. Assumptions, Open Questions, and Blockers
- ASSUMP-01: Simulator evidence is based on deterministic UI-test seeding, not on a live multi-device CloudKit publish/fetch loop between owner and invitee hardware.
- ASSUMP-02: The proposal review is scoped to iOS and does not assess Android or backend service parity.
- QUESTION-01: How should the new `rateSnapshotTimestamp` and tiered freshness metadata be versioned in local cache and CloudKit root/goal records so older cached payloads do not get misclassified?
- QUESTION-02: Should the detail screen expose the same relative freshness and rate-age semantics as the section header, or is the detail view intentionally allowed to use a different timestamp presentation?
- BLOCKER-01: Push-driven v2 behavior, multi-device owner races, and real network timing SLAs were not verified end-to-end in this evidence pass because the runtime evidence is simulator-seeded rather than CloudKit-live.
