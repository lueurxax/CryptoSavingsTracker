## H. Assumptions, Open Questions, and Blockers
- ASSUMP-01: The simulator evidence is seeded through repo-local UITest flags rather than a live multi-device CloudKit owner/invitee session.
- ASSUMP-02: This repeat pass evaluates proposal readiness against current runtime and docs; it does not claim implementation completeness.
- QUESTION-01: Should `contentHash` be defined over the full invitee-visible canonical projection payload, not only goal data, so owner display-name or participant-metadata updates cannot be deduplicated away?
- QUESTION-02: In the final locked header matrix, does `checkedNoNewData` append to the age-based primary freshness message or replace it? Section 6.4.2 and Section 8.2 currently point in different directions.
- QUESTION-03: Should the `materiallyOutdated` header matrix split publish-governed and rate-governed rows explicitly, matching the earlier grammar split, instead of using one `either` row?
- BLOCKER-01: No live two-owner-device CloudKit publish/import trace was captured for this repeat pass, so multi-device freshness races remain architecture-validated rather than runtime-replayed.
- BLOCKER-02: No proposed-state mockups or simulator captures exist yet for the new freshness-specific namespace header and detail Freshness card contract; the repo only contains older banner-first redesign previews, not this proposal's final visual system.
