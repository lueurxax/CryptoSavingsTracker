## D. Assumptions and Open Questions
- ASSUMP-01: No Android stale-draft resolution surface exists today in the planning UI; if there is an unreached flow elsewhere, it is not discoverable from the current repo search and should be called out before implementation.
- ASSUMP-02: Xcode previews are used as visual evidence for structure and affordances, not as a substitute for final simulator QA on iPhone/Pixel devices.
- ASSUMP-03: Workstream D is intended for goal create/edit forms only, not for the budget calculator sheet or other intentionally gated financial save surfaces.
- QUESTION-01: Should Workstream C keep resolved rows visible long enough to make the new state chips meaningful, or should the chip idea be dropped entirely in v1?
- QUESTION-02: Does the team want one generic copy-contract validator, or should planning copy keep a dedicated CI gate separate from Goal Dashboard artifacts?
- QUESTION-03: Is the intended implementation plan actually platform-split (iOS-heavy for B/C/D, parity-only for A), and if so should that be made explicit in the metadata and delivery plan?
